Read `AGENTS.md`, `README.md`, and `TODO.md` before making changes.

This repo uses Graphite. Do not use `git push`, `git checkout -b`, or `gh pr create`. Use `gt create` and `gt submit --stack` when branch/PR work is required.

This repo uses Linear for issue tracking. Keep the assigned Linear issue in scope, reference the issue identifier in branch/commit/PR context when appropriate, and avoid drifting into adjacent backlog work.

Preserve the template boundary. Prefer `packages/site-config`, `storefront/site`, and `backend/src/site` for client-specific behavior. Do not hardcode client branding into shared template paths when a delegation point is possible.

For storefront/backend changes, prefer small focused diffs. Run the narrowest relevant checks first, then the repo gates before handing off:

- `bun run typecheck`
- `cd storefront && bun run prettier:check && bun run test:unit`
- `cd backend && bun run prettier:check && bun run test:unit`
- `cd storefront && bunx next build --experimental-build-mode compile`

When touching checkout, cart, auth, newsletter, reviews, or other commerce-critical flows, preserve analytics and Sentry coverage. Do not add PII to analytics or error payloads.

GitHub remains the source of truth for PRs, CI, and review threads. Treat CodeRabbit feedback as GitHub review feedback when it appears on the PR. Before submitting or updating a PR, run `cr review` when available, fix valid findings, and resolve addressed review threads.

Do not overwrite or delete `.env` files or secrets. If local env-dependent verification is required, read existing repo docs first and fail safely.

Update docs when behavior materially changes, especially `README.md`, `TODO.md`, and feature docs under `docs/features/`.
