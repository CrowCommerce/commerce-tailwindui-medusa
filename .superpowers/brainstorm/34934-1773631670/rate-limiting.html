Rate Limiting — Failed-Attempt Tracking

Redis-backed middleware counting only failed auth attempts per IP

Why Not a Simple Rate Limiter?

You asked for "5 failed attempts → 15-minute lockout." A naive rate limiter counts ALL requests (including successes), which locks out legitimate users who typo once then succeed. Instead, we'll track failed attempts only using response status interception.

How It Works

Request Lifecycle
1
Pre-check: Is IP locked out?
Read auth_fail:{ip} from Redis. If count ≥ 5, return HTTP 429 with Retry-After header (TTL of key in seconds).
2
Let request proceed to Medusa auth handler
Call next() — Medusa handles the actual login/signup/reset logic.
3a
On 401 response → increment counter
Redis INCR auth_fail:{ip} + EXPIRE 900 (15 min). Uses res.on('finish') hook to inspect status after handler completes.
3b
On 200 response → clear counter
Redis DEL auth_fail:{ip}. Successful login resets the lockout window.

Graceful Degradation

If Redis is unavailable (no REDIS_URL, or connection fails), the middleware passes through — auth works normally without rate limiting. Logs a warning on startup. This ensures local dev without Redis is unaffected.

Storefront Error Handling

The existing login and signup server actions catch errors generically. When the backend returns 429, the Medusa SDK will throw a FetchError with status 429. We'll add a check in the catch block to surface a user-friendly message: "Too many login attempts. Please try again in 15 minutes."
